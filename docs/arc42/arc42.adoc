== 1. Introduction and Goals

=== 1.1 Requirements Overview

UDAL (Universal Device Abstraction Layer) provides a unified API surface for interacting with heterogeneous IoT devices regardless of their underlying transport protocol.

Core capabilities:

* *Device Registration* — devices declare their identity and capability schema on connect
* *Property Read / Write* — clients read and write typed device properties via a single API
* *Command Dispatch* — clients send named commands with typed parameters to devices
* *Streaming Telemetry* — clients subscribe to real-time property updates
* *Multi-Transport* — MQTT (3.1.1 + 5), HTTP (poll + webhook), CAN (SocketCAN) supported simultaneously

See link:../req42/req42.adoc[req42 Requirements Document] for the full requirement set.

=== 1.2 Quality Goals

[cols="1,2,3"]
|===
| Priority | Quality Goal | Scenario

| 1
| *Low Latency*
| A client reads a device property. End-to-end p99 latency (API request → adapter → device → response) is < 50 ms under normal load (≤ 1,000 devices, ≤ 100 req/s).

| 2
| *Protocol Transparency*
| A client application does not need to know or handle the device's underlying transport. The same API call works regardless of whether the device speaks MQTT, HTTP, or CAN.

| 3
| *Embedded Suitability*
| A device SDK (Rust) must run on a bare-metal Cortex-M microcontroller with < 8 KB flash and < 2 KB RAM overhead for the MQTT-only path.

| 4
| *Operational Simplicity*
| An operator deploys UDAL gateway as a single binary or Docker container with one config file. No external database required for v1.

| 5
| *Security by Default*
| All connections use TLS. Authentication is required on every request. A misconfigured or missing credential returns an explicit error, never silent pass-through.
|===

== 2. Architecture Constraints

=== 2.1 Technical Constraints

[cols="1,3"]
|===
| Constraint | Rationale

| Linux ≥ 5.10 for CAN adapter
| SocketCAN socket interface requires a modern kernel; Windows/macOS not supported for CAN in v1

| No mandatory external database
| v1 must run on edge hardware without a database server; bbolt embedded store used

| TLS 1.2 minimum everywhere
| Security requirement (NFR-004); plain-text mode only allowed in explicit `--dev` flag

| Protobuf as API contract
| Language-neutral; enforces schema; enables grpc-gateway for REST

| Single binary deployment
| Edge hardware (Raspberry Pi, industrial PC) may not have a container runtime
|===

=== 2.2 Organisational Constraints

[cols="1,3"]
|===
| Constraint | Rationale

| Apache 2.0 licence
| Permissive licence allows commercial use by end users

| Monorepo
| All components version-locked together; see ADR-001

| Conventional Commits
| Enables automatic changelog generation for releases

| No vendored dependencies
| Go modules, Cargo, pip, npm manage dependencies; `go mod vendor` not used
|===

== 4. Solution Strategy

[cols="1,2,2"]
|===
| Goal | Decision | Rationale

| Protocol transparency
| Pluggable `Transport` interface + Message Router
| Adapters are interchangeable; router resolves device → correct adapter at runtime

| Low latency
| In-process adapter calls (structured monolith)
| No IPC overhead; sub-millisecond routing from router to adapter

| Embedded suitability
| Rust SDK, no_std optional, MQTT-only path
| Rust zero-cost abstractions; MQTT needs no TLS handshake per property read

| Operational simplicity
| Single binary + bbolt embedded store
| No external DB; one config file; works on edge hardware and in containers

| Security by default
| TLS mandatory; three auth methods in interceptor chain
| Every deployment is secure out of the box; dev mode explicitly opt-in

| Open extension
| Transport interface as public API seam
| Third-party adapters can be compiled in without forking the gateway
|===

== 6. Runtime View

=== 6.1 Property Read

See `architecture.jsonc` dynamic view `property-read` for the sequence diagram.

. Client calls `GetProperty(deviceId, "temperature")` via gRPC
. Auth middleware validates credential and RBAC role
. Router resolves device transport config from Device Registry
. Router calls `ReadProperty()` on the MQTT Adapter
. MQTT Adapter publishes request topic; waits for device response (timeout: 5 s)
. Value returned back through router → API → client
. **Target:** p99 < 50 ms end-to-end

=== 6.2 Device Registration

See `architecture.jsonc` dynamic view `device-registration` for the sequence diagram.

. Device calls `Run(ctx)` in the Go SDK
. SDK calls `RegisterDevice(id, schema)` via gRPC with mTLS certificate
. Auth middleware validates mTLS certificate
. Router resolves schema in Capability Registry
. Router persists device entry in Device Registry
. SDK connects to MQTT broker and subscribes to command topics

=== 6.3 Telemetry Publish (Async)

See `architecture.jsonc` dynamic view `telemetry-publish` for the sequence diagram.

. Device publishes to MQTT topic `udal/{id}/props/temperature`
. MQTT Adapter receives event; forwards `PropertyEvent` to Router
. Health Monitor updates `last-seen` timestamp in Device Registry
. Router fans out event to all active `Subscribe` streams
. Each subscribed client receives `PropertyEvent` over gRPC server-side stream

=== 6.4 HTTP Property Read (F-10, issue #24)

See `architecture.jsonc` dynamic view `http-property-read` for the sequence
diagram. Same shape as §6.1, except step 4 onward: the HTTP Adapter has no
persistent connection or broker to reuse, so every `ReadProperty` is a
synchronous round trip against the device itself.

. Client calls `GetProperty(deviceId, "temperature")` via gRPC
. Auth middleware validates credential and RBAC role
. Router resolves device transport config from Device Registry
. Router calls `ReadProperty()` on the HTTP Adapter
. HTTP Adapter issues `GET {endpoint}/properties/temperature` against the
  device's `http.endpoint` label (timeout: 5 s)
. Value returned back through router → API → client

=== 6.5 HTTP Webhook Push (F-10, issue #24)

See `architecture.jsonc` dynamic view `http-webhook-push` for the sequence
diagram. Complements §6.4: a device can push a value ahead of the HTTP
Adapter's own background poll loop (poll interval configurable per device,
default 5 s — the poll loop follows the same fan-out path via its own
`PropertyEvent` call, not shown as a separate diagram).

. Device `POST`s to `{webhook_addr}/devices/{deviceId}/events` with
  `{"path": "temperature", "value": {...}}`
. HTTP Adapter validates the device is watched (`WatchDevice` was called at
  registration); forwards `PropertyEvent` to Router
. Router fans out event to all active `Subscribe` streams
. Each subscribed client receives `PropertyEvent` over gRPC server-side stream

=== 6.6 CAN Property Read (F-11, issue #25)

See `architecture.jsonc` dynamic view `can-property-read` for the sequence
diagram. Unlike §6.1/§6.4, step 4 onward isn't a live round trip at all: CAN
is a broadcast bus with no request/response semantics for an arbitrary
signal, so the CAN Adapter's read loop continuously decodes every frame
matching a known DBC message ID into an in-memory cache (independently of
any particular `GetProperty` call, potentially long before or after one),
and `ReadProperty` simply answers from it.

. Client calls `GetProperty(deviceId, "EngineSpeed")` via gRPC
. Auth middleware validates credential and RBAC role
. Router resolves device transport config from Device Registry
. Router calls `ReadProperty()` on the CAN Adapter
. CAN Adapter returns the signal's value from the last frame decoded for
  that device's DBC message (no bus access on this call)
. Value returned back through router → API → client

=== 6.7 CAN Property Write (F-11, issue #25)

See `architecture.jsonc` dynamic view `can-property-write` for the sequence
diagram. Unlike HTTP (§6.4/§6.5, read-only) and like MQTT, CAN supports
`WriteProperty` (issue #25's AC list, unlike #24's).

. Client calls `SetProperty(deviceId, "EngineTemp", value)` via gRPC
. Auth middleware validates credential and RBAC role
. Router resolves device transport config from Device Registry
. Router calls `WriteProperty()` on the CAN Adapter
. CAN Adapter encodes the value into the message's last-seen frame
  (read-modify-write, so sibling signals sharing the payload survive) and
  writes it to the SocketCAN interface
. CAN Adapter decodes its own just-written frame and fans out
  `PropertyEvent` to subscribers of that device — no round trip to a
  physical ECU is required to confirm the write, since the gateway itself
  produced the frame
. Router returns the new value back through API → client

== 8. Cross-cutting Concepts

=== 8.1 Error Handling

* All errors returned as gRPC status codes with structured `google.rpc.Status` details
* REST gateway translates gRPC status → HTTP status code automatically
* Adapter errors use circuit breaker (5 consecutive failures → open 30 s)
* Panics in adapter goroutines are recovered and logged; gateway continues

=== 8.2 Logging

* Structured JSON logs via `log/slog` (Go 1.21+, issue #28): `slog.NewJSONHandler`
  with its default `time` key renamed to `timestamp`, wrapped so a `trace_id`
  attribute is attached automatically to any log call made with a
  request-scoped `context.Context` (a `contextHandler` reading it back out —
  the documented `log/slog` pattern for context-carried attributes)
* Fields: `timestamp`, `level`, `component`, `trace_id`, `device_id`, `error` —
  `component` comes from a per-subsystem child logger
  (`log.With("component", "mqtt_adapter")` etc., one per adapter/subsystem);
  `trace_id` is present on request-scoped log lines only (see below)
* `UDAL_LOG_LEVEL` (`debug`\|`info`\|`warn`\|`error`) sets the level a
  freshly-started gateway begins at; `PUT /debug/log-level` on the metrics
  listener changes it live, since a running process can't observe its own
  environment changing without a restart
* One JSON line logged per gRPC request (also covering REST, proxied through
  the same server) by a `logging.Interceptor` running second in the chain,
  right after `tracing.Interceptor` and before auth — carries a per-request
  `trace_id`, the real OpenTelemetry `TraceID` of the `"api"` span
  `tracing.Interceptor` started (issue #29), read back out of `ctx` by
  `contextHandler`

=== 8.3 Observability

* **Metrics** (issue #27): Prometheus at `/metrics` — `udal_devices_online`,
  `udal_requests_total{operation,status}`, `udal_adapter_errors_total{adapter}`,
  `udal_request_duration_seconds{operation}`. `code/gateway/internal/metrics`
  package-level `promauto` collectors, recorded by a gRPC interceptor
  (requests/duration) and at `DeviceService`'s adapter call sites/
  `heartbeat.Monitor`'s status-transition hook (adapter errors/devices
  online, respectively) — served from `promhttp.Handler()`, no separate
  registry to keep in sync
* **Tracing** (issue #29): OpenTelemetry SDK, `code/gateway/internal/tracing`.
  `tracing.NewProvider` builds and globally registers a real
  `*sdktrace.TracerProvider` on startup — always, regardless of whether
  `UDAL_OTEL_ENDPOINT` is set, since its default sampler produces a real
  trace ID for every span even with no exporter attached (that's what keeps
  §8.2's per-request `trace_id` working when tracing export is "disabled").
  Only when `UDAL_OTEL_ENDPOINT` is non-empty is an OTLP/gRPC exporter
  (`otlptracegrpc`) attached via `WithBatcher`; a bare `host:port` value
  dials plaintext, a full URL (`https://...`) dials TLS. Span tree per
  request: `api` (`tracing.Interceptor`, first in the gRPC interceptor
  chain) → `auth` (`auth.Authenticator`) as a sibling child, and → `router`
  (`DeviceService.GetProperty`/`SetProperty`) → `adapter` (mqtt/http/can
  dispatch) as a nested child, only for those two RPCs and only when routed
  to a transport adapter. W3C Trace Context propagation
  (`propagation.TraceContext`/`Baggage`) is registered but not yet exercised
  by any outbound call this gateway makes.
* **Health** (issue #27): `GET /health` returns HTTP 200 `{"status":"ok"}` when the gateway is ready, `503 {"status":"starting"}` before every listener is up, and includes a per-adapter `"degraded"` status (still HTTP 200) for any adapter implementing `health.Reporter` (MQTT: circuit breaker open; CAN: read loop stopped) — served on the same metrics listener as `/metrics` and `/debug/log-level`

=== 8.4 Configuration

Single YAML file with full env-var override (`UDAL_` prefix):

[source,yaml]
----
gateway:
  grpc_port: 50051
  http_port: 8080
  metrics_port: 9090
  tls:
    cert: /etc/udal/tls/server.crt
    key:  /etc/udal/tls/server.key
    ca:   /etc/udal/tls/ca.crt
  auth:
    api_key_header: X-API-Key
    jwks_url: https://auth.example.com/.well-known/jwks.json
  registry:
    path: /var/udal/registry.db
  adapters:
    mqtt:
      broker: tcp://mosquitto:1883
      client_id: udal-gateway
    http:
      poll_interval: 5s
      webhook_port: 8090
      mtls:
        cert: /etc/udal/tls/http-adapter-client.crt
        key:  /etc/udal/tls/http-adapter-client.key
    can:
      interface: can0
      dbc_file: /etc/udal/can/vehicle.dbc
  heartbeat_interval: 30s
  device_timeout: 90s
----

=== 8.5 Security

* TLS mandatory on all listeners (opt-out via `--dev` flag only)
* Auth interceptor runs before any handler; no handler is reachable unauthenticated
* API keys stored as bcrypt hashes; never returned in API responses
* mTLS: gateway validates client certificate chain against configured CA
* Input validation on all gRPC message fields; invalid values → `INVALID_ARGUMENT`

== 10. Quality Requirements

=== 10.1 Quality Scenarios

[cols="1,2,3,2"]
|===
| ID | Stimulus | Response | Measure

| QS-01
| Client calls GetProperty under 100 req/s with 1,000 connected MQTT devices
| Gateway routes request, adapter queries device, returns value
| p99 latency < 50 ms

| QS-02
| MQTT broker becomes unreachable
| MQTT adapter activates circuit breaker; CAN and HTTP adapters continue; MQTT devices return UNAVAILABLE
| Other adapters respond normally within 1 s of broker failure

| QS-03
| Attacker sends request without any credential
| Auth middleware rejects immediately with UNAUTHENTICATED; no device state touched
| 100% of unauthenticated requests rejected; no information leak

| QS-04
| Operator deploys gateway binary on Raspberry Pi 4
| Gateway starts, connects to Mosquitto, serves API
| Start-up time < 3 s; RAM usage < 150 MB at idle

| QS-05
| Rust SDK embedded in bare-metal Cortex-M firmware
| SDK compiles with no_std, registers device, publishes property
| Flash overhead < 8 KB; RAM overhead < 2 KB; no heap allocator required

| QS-06
| New contributor adds a third-party CAN J1939 adapter
| Contributor implements Transport interface, compiles it into the gateway
| Zero changes to gateway core; CI passes; adapter discoverable via config

== 11. Risks and Technical Debt

[cols="2,1,2,2"]
|===
| Risk | Probability | Impact | Mitigation

| CAN adapter panic brings down entire gateway (structured monolith)
| Medium
| High — all devices affected
| recover() around each frame's decode in the adapter's own read-loop goroutine (issue #25) — one bus's malformed/unexpected frame can't crash the process. No circuit breaker: unlike MQTT there's no persistent request/response connection to protect (same reasoning as the HTTP adapter, #24) — the read loop just keeps draining the socket regardless of any one frame's outcome.

| bbolt registry performance degrades with > 10,000 devices
| Low (v1 scope: 1,000)
| Medium — higher latency on ListDevices
| Add pagination to ListDevices; PostgreSQL backend planned for v2

| MQTT broker as single point of failure for MQTT devices
| Medium
| High for MQTT-only deployments
| Document broker HA setup (EMQX cluster); future: built-in MQTT broker option

| Rust SDK no_std breakage on RTOS updates
| Low
| Medium — embedded targets affected
| CI test on ARM cross-compilation target; maintain QEMU test for bare-metal path

| Breaking proto API changes without clients noticing
| Medium
| High — all SDK consumers affected
| buf breaking check in CI; semantic versioning; deprecation policy before removal
|===

== 12. Glossary

[cols="1,3"]
|===
| Term | Definition

| Adapter
| A gateway-internal plugin implementing the `Transport` interface for one protocol

| bbolt
| Embedded key/value store for Go (formerly boltdb); used for Device and Capability registries

| buf
| Protobuf toolchain for linting, formatting, and breaking-change detection

| Capability Schema
| YAML/JSON document describing a device's properties, commands, and events

| CAN
| Controller Area Network — automotive/industrial fieldbus protocol

| Circuit Breaker
| Pattern: after N consecutive errors, stop calling a failing dependency for a cooldown period

| gRPC
| Google Remote Procedure Call — binary RPC over HTTP/2 using protobuf

| grpc-gateway
| Go library that generates REST/JSON proxy from gRPC service definitions

| MQTT
| Message Queuing Telemetry Transport — lightweight pub/sub protocol for IoT

| no_std
| Rust compilation mode that excludes the standard library; suitable for bare-metal targets

| RBAC
| Role-Based Access Control — permissions assigned to named roles

| Reflex
| Python full-stack web framework that compiles Python to React/Next.js

| SocketCAN
| Linux kernel subsystem providing a socket-based API for CAN bus access

| Transport
| Go interface (`ReadProperty`, `WriteProperty`, `SendCommand`, `Subscribe`) implemented by each adapter

| UDS
| Unified Diagnostic Services (ISO 14229) — automotive ECU diagnostic protocol
|===
